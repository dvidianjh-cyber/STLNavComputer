# STL Nav Computer v0.1.1 Bugs & Updates

## Context
**Project:** STL Nav Computer
**Target Version:** 0.1.1 -> 0.1.2
**Objective:** Address identified bugs in the modal/window system, fix the Journey Planner layout and functionality, and adjust global theme contrast.

## 1. Bug Fixes
* **Movable Window Close Action:** * *Current State:* Clicking an informational modal correctly converts it into a movable window, but the "Close" button on the new movable window is unresponsive.
  * *Required Action:* Bind the close button event to properly unmount/destroy the movable window component from the DOM. Ensure associated state variables tracking the window's visibility are reset.

## 2. Feature Enhancements (Journey Planner)
* **Collapsible Journey Planner:**
  * *Requirement:* Add a small toggle button (collapse/expand icon) in the top-right corner of the Journey Planner container.
  * *Behavior:* Clicking the button must collapse the main body of the Journey Planner.
  * *Collapsed State:* Only the Journey Planner title and the collapse/expand toggle button should remain visible.
* **Multi-Waypoint Layover Layout Correction:**
  * *Current State:* The "Layover days" input/layout is globally positioned or in the wrong component hierarchy.
  * *Required Action:* Refactor the layout so that a specific "Layover days" input field is dynamically generated and bound to *each individual stop* within a multi-waypoint flight plan array. Update the state management and total journey calculations to handle layover durations per waypoint index.

## 3. UI and Theme Adjustments
* **Global Typography Contrast:**
  * *Required Action:* Increase the brightness/lightness (e.g., higher lightness value in HSL or closer to `#FFFFFF` in Hex) of the main text font to improve readability.
* **Global Background Contrast:**
  * *Required Action:* Darken the primary blue background color to improve the text-to-background contrast ratio.

## 4. Feature: Movable Window Origin Tether
* **Requirement:** Draw a visual connector (a thin line) linking a movable info window to the specific star it describes.
* **Constraints & Logic:**
  * **Condition:** This line must *only* render for windows that have been converted into the "movable" state (detached from their initial modal state).
  * **Dynamic Tracking:** The line must anchor to the 2D coordinates of the origin star and the current 2D coordinates of the movable window (e.g., the closest edge or the center of the window).
  * **Event Listeners:** The line's rendering coordinates must update synchronously during `onDrag` events for the window, as well as `onPan` or `onZoom` events for the main navigational map/canvas.
  * **Rendering Implementation:** Depending on the current map rendering context (Canvas, WebGL, or DOM/SVG), ensure coordinate spaces are properly translated (e.g., world space to screen space) so the line does not detach visually.

## 5. Feature: Flight Plan Leg Labels
* **Requirement:** Display the "subjective journey time" for each individual leg of a multi-waypoint flight plan directly on or adjacent to the drawn trajectory lines.
* **Constraints & Logic:**
  * **Data Mapping:** Iterate through the `waypoints` or `legs` array resulting from the journey calculation. Extract the specific subjective time for *each leg*, rather than the total journey time.
  * **Positioning:** Calculate the midpoint of the line segment for each leg `((x1 + x2) / 2, (y1 + y2) / 2)` to anchor the text label.
  * **Legibility:** * Add a slight orthogonal offset to the text position so it does not overlap and obscure the trajectory line itself.
    * Render the text using a small, readable font size. Include a subtle text-shadow or background pill to ensure contrast against the star map background.
  * **Redraw Trigger:** Ensure these labels are cleared and redrawn whenever the journey is recalculated or the map view is altered.

# Agent Instructions & Edge Cases to Handle
* **Z-Index / Layering:** Ensure the tether lines are rendered on a layer *below* the UI windows but *above* the background map layer so they do not obscure text.
* **Math Precision:** When calculating the midpoint for the journey time labels, handle potential division-by-zero or floating-point inaccuracies if two waypoints occupy the exact same coordinate space (e.g., a zero-distance jump). Hide the label if the distance is 0.
* **Component Cleanup:** When fixing the movable window bug, ensure proper garbage collection/unmounting to prevent memory leaks if multiple modals are opened and closed.
* **State Management:** For the multi-waypoint layovers, ensure the data schema reflects an array of objects (e.g., `waypoints: [{ id, destination, layoverDays }, ...]`).
* **CSS Consistency:** Apply color changes using the existing CSS variables or theme provider system if one is present in the codebase.
