# Technical Specification: Nav-Computer v0.2.0 Update

## 1. Context and Objective
This document outlines the requirements for version 0.2.0 of the "Nav-Computer" STL Navigation System. The update requires architectural changes to the JSON data schema, resolution of a CSS/DOM rendering bug, and the introduction of new UI elements and Three.js map features. 

Target Stack: Vanilla HTML5, CSS3, ES6 Modules, Three.js (r165).

---

## 2. Data Architecture Changes
The `systems.json` structure is being expanded to encapsulate metadata. 

### 2.1 Schema Modification
* **Action:** Wrap the existing array of star systems into a `systems` node and introduce root-level metadata.
* **Expected New Schema:**
    ```json
    {
      "map_name": "Ondari Empire Local Space",
      "map_description": "Standard exploratory stellar neighborhood",
      "systems": [
        { "name": "Sol", "type": "Yellow Dwarf", "coords": [0,0,0], ... }
      ]
    }
    ```
* **Agent Directives:** * Update `dataLoader.js` (or equivalent) to extract the data array from the `systems` key.
    * Store `map_name` and `map_description` in the application state for UI consumption.

---

## 3. UI Modifications & Additions

### 3.1 Title Panel
* **Action:** Remove the hardcoded text from the "STL NAVIGATION SYSTEM" and "X systems charted" labels.
* **Action:** Instead Bind the text to the newly fetched `map_name` (as the subtitle, instead of "STL NAVIGATION SYSTEM") and `map_description` (instead of "X Systems Charted").  Keep the "NAV-COMPUTER" primary title.

### 3.2 "Current System" Panel (New Feature)
* **Location:** Top Center of the viewport.
* **Components:**
    1.  **Dropdown Menu (`<select>`):** Populated dynamically with all system names.
    2.  **Center View Button/Icon:** Centers the Three.js viewport/camera on the currently selected system.
* **Interaction Logic:** Selecting a system updates the global state for "Selected System", which must trigger the 3D Highlight Sphere (see section 4.1).

### 3.3 Map Key Panel (Bottom Left)
* **Dynamic Generation:** Do not hardcode system types. Derive the list of system types dynamically by iterating through the loaded `systems` array and extracting unique `type` string values.
* **Label Toggles:** Add a checkbox/toggle switch next to *each* dynamically generated system type to control the visibility of their respective 3D text labels.
* **Collapsibility:** Implement an expand/collapse toggle function utilizing the same CSS/JS methodology currently used by the Journey Planner panel.

### 3.4 Journey Planner
* **Default State:** Ensure the UI panel initializes in a collapsed state.
* **Drive Output Input:** Convert the input from a standard number field to a `<input type="range">` slider. Max value: `10` (representing Gs).
* **Layout Adjustment:** Place the new Drive Output Slider and the existing Max Cruise Velocity input on the same row.
    * **Constraints:** Use a flex/grid layout where the Slider occupies `70%` of the width, and the Max Cruise Velocity input occupies `30%` of the width.

### 3.5 Pinned Info Windows
* **Action:** Remove the System `id` property from the rendered UI of the pinned/movable data cards.
* **Bug Fix (The "Flash" Issue):** * **Symptom:** Cards briefly flash on the left side of the screen upon creation before jumping to the pointer/hover coordinates.
    * **Agent Directive:** This is a lifecycle/paint issue. Inject the new DOM element with `visibility: hidden` or `opacity: 0`. Only transition to `visibility: visible` or `opacity: 1` *after* calculating and applying the inline `top` and `left` pixel coordinates.

---

## 4. Three.js Renderer & Map Updates

### 4.1 Selected System Highlight
* **Action:** The blue semi-transparent highlight sphere must no longer default to the origin `(0,0,0)`.
* **Behavior:** The sphere's position must actively bind to the coordinates of the system selected in the new "Current System" dropdown.

### 4.2 Dynamic Map Grid
* **Action:** The 2D planar grid helper on the XZ plane must scale dynamically based on the dataset.
* **Calculation:** During data parsing, calculate the maximum absolute distance from the origin on the X and Z axes. Set the Three.js `GridHelper` size to encompass this maximum distance plus an arbitrary 10% padding boundary.

### 4.3 3D System Labels (New Feature)
* **Action:** Render text labels representing the system names directly in the 3D viewport next to their respective star nodes.
* **Agent Directives:**
    * Use either `CSS2DRenderer` for crisp, UI-matching HTML labels, or Three.js `Sprite` textures if performance is a strict bottleneck (prefer `CSS2DRenderer` for text clarity).
    * **Filtering Logic:** Bind the visibility of these labels to the toggles in the Map Key Panel. If the 'Unexplored' toggle is off, all `CSS2DObject` elements associated with 'Unexplored' systems must have their visibility set to `false`.
    * Ensure labels smoothly follow the camera during `OrbitControls` rotation.

---

## 5. Development Constraints
* **No Build Tools:** Maintain the zero-build-step architecture. Do not introduce Webpack, Vite, or TypeScript. Stick to ES6 module imports.
* **Backward Compatibility:** Ensure `physics.js` properly accepts the newly nested coordinate data without breaking the relativistic turnover calculations.
* **Robust State Management:** Centralize the `selectedSystem`, `hiddenLabelTypes`, and `cameraTarget` variables in `app.js` to ensure sync between the DOM overlay (`ui.js`) and the WebGL canvas (`renderer.js`).
