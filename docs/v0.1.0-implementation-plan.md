#### [NEW] `js/physics.js`

The relativistic engine — pure functions, no DOM:

```js
// Constants
const C = 1.0;           // speed of light in LY/Year
const G_IN_LY_PER_Y2 = 1.03;  // 1G in LY/Year²
const V_MAX_ABS = 0.9999999;   // absolute cap

/**
 * @param {number} gForce - in Gs
 * @param {number} maxCruiseFraction - 0–0.9999999
 * @param {number} distanceLY
 * @returns {{ tObj, tSubj, phases: [{type, fraction}] }}
 */
export function calculateLeg(gForce, maxCruiseFraction, distanceLY) { ... }

export function calculateDistance(a, b) { ... }  // 3D Euclidean
export function calculateRoute(waypoints, gForce, maxCruise, layoverDays) { ... }
```

**Key physics algorithm per leg (rest-to-rest):**
1. Compute acceleration `a = gForce × 1.03 LY/Y²`
2. Compute `v_cruise = min(maxCruiseFraction × c, V_MAX_ABS × c)`
3. Compute `d_acc` = distance to reach `v_cruise` under constant thrust: `d_acc = v²/(2a)`
4. **Turnover catch:** if `2 × d_acc ≥ d`, skip cruise phase → find mid-point turnover velocity, recalculate
5. Compute objective and subjective time for each phase using relativistic rapidity integrals
6. Return `phases[]` with `{type: 'acc'|'cruise'|'dec', fraction: 0-1}` for the renderer

#### [NEW] `js/renderer.js`

Manages the entire Three.js scene:

- `initScene(canvas)` — scene, PerspectiveCamera, WebGLRenderer, OrbitControls, ambient + point lights
- `loadStars(systems)` — creates `THREE.Mesh` spheres per system, color by type, with a subtle emissive glow
- Background: large inverted `THREE.SphereGeometry` with a dark starfield texture (generated procedurally from a canvas 2D context — no external image needed)
- `drawRoute(waypoints, phases[])` — builds a `THREE.TubeGeometry` or `Line` per segment, colored by phase
- `clearRoute()` — disposes old geometries
- **Render loop:** uses `requestAnimationFrame` but only calls `renderer.render()` if `needsRender` flag is true (dirty-flag pattern for performance)
- Exports `markDirty()` called by interaction/UI when redraw is needed

---

#### [NEW] `js/interaction.js`

- Throttled `mousemove` handler (16ms / ~60fps cap via `setTimeout` gate)
- `THREE.Raycaster` against the star mesh array
- On hover: sets emissive highlight, calls `ui.showHoverCard(system, screenPos)`
- On click: calls `ui.pinCard(system, screenPos)` and marks dirty
- Drag logic for pinned cards: `pointerdown` → `pointermove` → `pointerup` using `transform: translate()`, constrained to `window.innerWidth/Height`

---

#### [NEW] `js/ui.js`

- `initUI(systems)` — populates `<datalist>` or select elements from the loaded system array
- `showHoverCard(system, pos)` / `hideHoverCard()` — transient tooltip card
- `pinCard(system, pos)` — clones hover card, makes it draggable, adds close button
- `displayResults(routeResult)` — writes objective time, subjective time, total distance to results strip
- `resetResults()` — clears to "Awaiting Input..." state
- `setLoading(bool)` — shows/hides a spinner overlay during data fetch

---

#### [NEW] `js/app.js`

Central orchestrator and single source of truth:

```js
// State
const state = {
  systems: [],
  selectedWaypoints: [],    // array of system IDs
  gForce: 1.0,
  maxCruise: 0.99,
  layoverDays: 0,
  currentRoute: null,
};
```

- Bootstraps on `DOMContentLoaded`
- Calls `dataLoader → renderer.loadStars → interaction.init → ui.initUI`
- Listens for "CALCULATE ROUTE" click → validates waypoints → calls `physics.calculateRoute` → calls `renderer.drawRoute` → calls `ui.displayResults`
- All state mutations flow through `app.js`; modules receive data, not state references
